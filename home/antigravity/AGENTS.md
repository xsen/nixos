# Глобальные правила Antigravity

- **Язык общения:** Всегда отвечай пользователю на русском языке.
- **Язык разработки:** Любой генерируемый код, технические комментарии (включая док-блоки), логи, системные сообщения и коммиты пиши строго на английском. **Исключение:** Тексты пользовательского интерфейса (UI, кнопки, надписи, плейсхолдеры в HTML/JSX/Vue) и файлы локализации (i18n, json-локали) пиши на языке целевой аудитории проекта (по умолчанию на русском, если проект ориентирован на русскоязычных пользователей, либо на языке целевой локали проекта).

## Браузерная автоматизация и MCP

- **Приоритетный браузер**: Для любых задач по автоматизации, тестированию или отладке веб-страниц по умолчанию используй инструменты **Chrome DevTools** (плагин `chrome-devtools-mcp`). Инструменты Playwright используй только при явном указании пользователя.
- **Особенности окружения (NixOS)**: Запуск Chrome DevTools MCP и Playwright MCP осуществляется через специальные скрипты-обертки, расположенные в каталоге плагина.

### Режимы запуска и Переменные окружения

При запуске браузера учитывай следующие переменные окружения, которые пользователь может передавать в CLI:

#### Для Chrome DevTools (приоритетный):

- `AGY_CHROME_HEADLESS` (по умолчанию `true`): если установлено в `false`, браузер запустится с отображением графического окна (headed).
- `AGY_CHROME_ISOLATED` (по умолчанию `true`): если установлено в `false`, используется постоянный профиль пользователя в `~/.config/chrome-mcp-profile` (сохраняет сессии и авторизации на веб-ресурсах). По умолчанию (`true`) используется временный изолированный профиль.

#### Для Playwright (резервный):

- `AGY_PLAYWRIGHT_HEADLESS` (по умолчанию `true`): если `false`, запускает headed-режим.
- `AGY_PLAYWRIGHT_ISOLATED` (по умолчанию `true`): если `false`, использует постоянный профиль в `~/.config/playwright-mcp-profile`.

_Пример запуска для ручной авторизации на сайтах (headed + non-isolated):_
`AGY_CHROME_HEADLESS=false AGY_CHROME_ISOLATED=false agy`

### Динамическое управление режимами по запросу пользователя

Если пользователь просит запустить браузер в специфическом режиме в процессе диалога (например, _"проверь через хром не в изолированном режиме с показом окна"_):

1. **Для Chrome DevTools (предпочтительно)**:
   - Не полагайся на дефолтный запуск MCP. Вместо этого принудительно запусти браузер самостоятельно в фоновом процессе через `run_command` с нужными флагами:
     - Оконный режим (headed): без флага `--headless`
     - Неизолированный режим (non-isolated): `--user-data-dir=/home/evgeny/.config/chrome-mcp-profile`
     - Порт отладки: `--remote-debugging-port=9222`
   - После этого вызывай MCP-инструменты `chrome-devtools`. Локальный скрипт плагина автоматически подключится к уже запущенному тобой инстансу Chrome на порту 9222.

2. **Для Playwright / Сложных сценариев**:
   - Напиши временный Node.js скрипт автоматизации в директорию `scratch` (используя пакет `playwright` из кэша `/home/evgeny/.npm/_npx/9833c18b2d85bc59/node_modules`), настроив в объекте `chromium.launch` любые параметры по запросу пользователя. Запусти его через `run_command`.

## Workflow проверки кода (Code Review Workflow)

- Для любых нетривиальных изменений кода (создание новых функций, рефакторинг, изменение логики конфигурации) перед сдачей задачи пользователю ОБЯЗАТЕЛЬНО используй навык [code-review](file:///home/evgeny/.gemini/config/skills/code-review/SKILL.md).
- Перед запуском авто-ревью убедись, что все текущие изменения сохранены на диск.
- Всегда запускай изолированного субагента-цензора (`invoke_subagent` с ролью `Senior Code Reviewer` и `TypeName: "self"`), чтобы получить непредвзятую оценку изменений в чистом контексте.
- Внимательно изучай отчет субагента и исправляй найденные критические ошибки перед тем, как завершить выполнение задачи.
- **Исключения**:
  1. Если изменения тривиальны (затрагивают только 1 файл и содержат менее 10 строк изменений, например, исправление опечатки, комментария или мелких констант).
  2. Если пользователь явно просит применить изменения без ревью (используя фразы вроде «без ревью», «без проверки», «быстро», «skip review»), пропусти этот шаг и заверши задачу напрямую.

## Экономия токенов и Качество: Паттерн "Plan-First" (Сначала План)

- Для задач, затрагивающих более 2 файлов или содержащих сложные архитектурные изменения, перед написанием кода ОБЯЗАТЕЛЬНО составь краткий текстовый план (что и где будет изменено).
- Не приступай к написанию кода, пока пользователь не подтвердит план (словами «ОК», «поехали» или «делай» ). Это предотвращает генерацию неверного кода и экономит ваши токены.
- **Динамическое перепланирование (Re-planning)**: Если в процессе выполнения плана возникает критическая ошибка сборки, линтинга или тестирования, остановись. Не пытайся «вслепую» чинить её мелкими правками. Сделай шаг назад, проанализируй ошибку, обнови план (добавь шаги исправления) и продолжай работу по скорректированному плану.
- **Лимит на автоисправления (Экономия бюджета)**: Максимальное число попыток автоисправления одной и той же ошибки сборки/тестирования — **2 раза**. Если после двух попыток исправления и перепланирования ошибка сохраняется, не трать токены на циклы — немедленно остановись, опиши проблему и обратись к пользователю.
