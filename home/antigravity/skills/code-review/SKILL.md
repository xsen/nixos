# Навык: Автоматическое ревью кода (code-review)

Этот навык используется основным агентом для проверки корректности внесенных изменений перед тем, как представить результат пользователю. Навык автоматически адаптируется под стек текущего проекта.

## Алгоритм работы:

1. **Анализ стека проекта**:
   - Быстро проанализируй проект, чтобы понять стек технологий (например, NixOS, Go, Rust, Node.js/TypeScript, Python) и доступные инструменты тестирования/сборки (наличие `package.json`, `Cargo.toml`, `flake.nix`, `pytest`, `eslint` и т.д.).
   - **Важно**: Убедись, что все текущие изменения сохранены на диск (`write_to_file` / `replace_file_content`).

2. **Подготовка диффа и контекста (Экономия токенов!)**:
   - Выполни команду `git add -N .` (если репозиторий существует), чтобы учесть новые (untracked) файлы в диффе.
   - **Не передавай файлы целиком**. Используй команду `git diff -U5` для получения изменений с компактным контекстом (5 строк вокруг изменений).
   - **Фильтруй шум**: Принудительно исключай из диффа файлы блокировок, зависимости и автогенерируемый код. Используй маски исключений, например:
     `git diff -U5 -- . ':(exclude)*lock*' ':(exclude)*node_modules*' ':(exclude)vendor*' ':(exclude)*.min.js' ':(exclude)dist*'`
   - Если дифф превышает 100 строк, запиши его во временный файл в директории `scratch/` (например, `scratch/review_diff.patch`) и передай субагенту ссылку на файл. Если дифф меньше, передай его текст прямо в Prompt.

   - Подготовь список команд для тестирования или сборки, которые субагент должен попытаться запустить.

3. **Запуск цензора**:
   - Создай изолированного субагента с помощью инструмента `invoke_subagent`.
   - **TypeName**: `self`
   - **Role**: `Senior Code Reviewer`
   - **Prompt**:
     ```text
     Ты — опытный Senior Code Reviewer. Твоя задача — провести непредвзятое и строгое ревью кода.
     
     Стек проекта: [Укажи стек, например: NixOS / Go / Node.js]
     Внесенные изменения (файлы): [Список файлов или ссылка на них]
     Дифф изменений: [Вставь текст диффа ИЛИ ссылку на scratch/review_diff.patch]
     
     Выполни следующие шаги в своем изолированном контексте:
     1. Изучи изменения в коде. Разделяй замечания по следующим приоритетам:
        
        - **1. Critical (Must Fix):**
          * Безопасность (утечки секретов, инъекции, небезопасные права доступа).
          * Логические ошибки, баги и крайние случаи (edge cases), приводящие к сбою.
          * Нарушения целостности данных или утечки ресурсов.
        
        - **2. Important (Should Fix):**
          * Производительность (неэффективные циклы, N+1 запросы, OOM).
          * Читаемость, избыточная сложность функций, неудачное именование.
          * Отсутствие обработки ошибок в критических местах.
        
        - **3. Style (Nice to Fix):**
          * Соответствие стилю проекта, опечатки, устаревшие комментарии.
     
     2. Попробуй запустить сборку, линтер или тесты с помощью доступных инструментов (например, запусти: [Вставь сюда реальные команды сборки/тестирования, которые ты определил на шаге 2, например: nix-build или npm run test]).
     
     3. Напиши структурированный отчет в следующем формате:
        
        ### Отчет о ревью кода
        **Вердикт**: [ОДОБРЕНО / ТРЕБУЕТ ДОРАБОТКИ]
        
        #### Замечания по файлам:
        *Для каждого файла с замечаниями:*
        * **[Имя файла](путь к файлу)**:
          * **[Тип замечания (Critical/Important/Style)]** (строки X-Y):
            - *Описание проблемы*: Объясни «почему» это является проблемой.
            - *Рекомендация*: Предложи конкретный вариант исправления кода.
        
        #### Результаты сборки/тестирования:
        - [Успешно / Ошибки с логами]
        
        *Негативные ограничения:*
        - Не запускай процессы в фоне, не копируй системные бинарники и не выполняй интеграционные тесты, которые могут потребовать ручного одобрения пользователя (такие как cp, rm, фоновые sleep). Ограничивайся быстрой сборкой/проверкой синтаксиса и статическим анализом (dry run).
        - Не ругай за стиль, если он соответствует остальному проекту.
        - Не предлагай рефакторинг, если текущий код прост и эффективен.
        - Твоя цель — дать фидбек, не пытайся сам исправлять файлы в основной ветке.
     ```

4. **Обработка фидбека**:
   - Получив отчет от субагента, проанализируй его.
   - Если есть критические замечания (Critical/Important), внеси исправления в код.
   - При необходимости повтори процесс (запусти ревью еще раз).
   - Если код одобрен или замечания некритичны и исправлены, представь результат пользователю, упомянув, что код прошел внутреннее ревью.
